# DB ARCHITECTURE

---

## 1. 아키텍처 개요 (Overview)

K-ETS 가격 예측 모델의 안정적 데이터 공급 및 피처(Feature) 도출을 위해 구성된 PostgreSQL 데이터베이스 아키텍처입니다. 
현재 DB는 원천 데이터 보존을 우선하여 `raw`(원천 데이터), `dim`(정적 차원 데이터), `meta`(문서/출처/품질 catalog)를 유지합니다. `label` 스키마는 타겟/라벨 설계 확정 전까지 생성하지 않습니다.

| 스키마명 | 역할 | 주요 구성 요소 | 비고 |
| --- | --- | --- | --- |
| **`raw`** | 원천 데이터 원본 유지 (불변) | 32개 원천 시계열/명세 데이터 | `메타데이터.md`와 1:1 매핑. P2/P3 EPSIS/KMA external raw 6개 포함. Legacy placeholder `kpx_daily`, `kma_weather_daily`는 0행·정본 CSV 없음으로 삭제 |
| **`dim`** | 분석 기준이 되는 차원(Dimension) 데이터 | `period`(계획기간), `vintage`(할당연도) | 정적 시드 데이터 |
| **`meta`** | 문서/출처/컬럼/공시/품질 catalog | `source_catalog`, `table_catalog`, `column_catalog`, `disclosure_rule_catalog`, `quality_rule_catalog` | raw/dim 해석용 metadata. target/feature/model output은 포함하지 않음 |
| **`label`** | 파생 변수 및 타겟(Target) 라벨링 | - | 현재 미생성. 타겟 설계 확정 후 재생성 |

## 3. DIM 스키마 명세 (`dim.*`)

배출권 제도의 구조적 특징(계획기간, 할당연도)을 정의하여 타 테이블의 기준으로 사용됩니다.

### 3.1 `dim.period` (계획기간)

| 컬럼명 | 타입 | 제약조건 | 설명 | 비고 |
| --- | --- | --- | --- | --- |
| `period_id` | SMALLINT | PRIMARY KEY | 계획기간 식별자 | 1, 2, 3, 4 |
| `start_year` | SMALLINT | NOT NULL | 시작 연도 | |
| `end_year` | SMALLINT | NOT NULL | 종료 연도 | |
| `description` | TEXT | | 기간에 대한 상세 설명 | 제1차 계획기간 등 |

### 3.2 `dim.vintage` (빈티지/할당연도)

| 컬럼명 | 타입 | 제약조건 | 설명 | 비고 |
| --- | --- | --- | --- | --- |
| `vintage_year` | SMALLINT | PRIMARY KEY | 할당연도 (Vintage) | 2015 ~ 2030 |
| `period_id` | SMALLINT | FK (dim.period) | 속한 계획기간 식별자 | |
| `is_active` | BOOLEAN | DEFAULT true | 활성 거래 연도 여부 | |

## 4. META 스키마 명세 (`meta.*`)

`meta` 스키마는 원천/차원 데이터의 의미와 검증 규칙을 DB 안에서 추적하기 위한 catalog 계층입니다. row-level 원천 데이터나 모델 산출물을 저장하지 않으며, `raw`/`dim` 테이블을 해석하기 위한 metadata만 보관합니다.

| 테이블명 | 역할 | 주요 키 | 비고 |
| --- | --- | --- | --- |
| `source_catalog` | 원천 출처 정의 | `source_id` | KRX, GIR, EPSIS, KMA ASOS, 공공기관, 내부 기준 차원 |
| `table_catalog` | live `raw`/`dim` 테이블 catalog | `schema_name`, `table_name` | row count, column count, 주기, source, external raw 여부 |
| `column_catalog` | live DB column catalog와 문서 컬럼 설명 연결 | `schema_name`, `table_name`, `column_name` | DB data type, nullable, 문서 설명/단위 |
| `disclosure_rule_catalog` | 공시 lag 및 leakage-safe 사용 규칙 | `rule_id` | external raw 6개는 ontology quality rule과 연결 |
| `quality_rule_catalog` | row count, PK/UNIQUE, ontology 품질 rule | `rule_id` | target/feature/model output 검증은 포함하지 않음 |

적재 스크립트는 `scripts/db_load_meta_catalog.py`이며, 기본 실행은 dry-run으로 `outputs/tables/meta_catalog/` 아래 CSV/JSON을 생성합니다. PostgreSQL 적재는 명시적 `--execute` 옵션을 사용합니다. 검증 스크립트는 `scripts/db_verify_meta_catalog.py`입니다.

## 5. LABEL 스키마 명세 (`label.*`)

현재 DB에는 `label` 스키마가 없습니다. 기존 `label.target_returns` 물질화뷰와 `label.policy_events` 테이블은 타겟/정책 이벤트 설계가 확정되기 전에 생성된 객체였으므로 2026-05-06에 삭제했습니다.

향후 라벨 계층은 다음 원칙에 따라 재설계합니다.

- `raw.*`는 원천 데이터만 보존합니다.
- 타겟 변수, 수익률, 변동성, 정책 이벤트 등 모델 종속 객체는 설계 확정 후 `label` 또는 별도 mart/feature 계층에 생성합니다.
- VWAP 기반 타겟, 대표 vintage 선정, 기세일/저유동성 처리 정책은 `docs/reference/가격_모델.md`와 `docs/plans/feature-candidates.md` 논의가 끝난 뒤 DDL로 확정합니다.
- 확정 전 파생변수는 DB에 영구 적재하지 않고 분석 스크립트나 parquet 산출물에서 실험합니다.
- `docs/ontology/allowance.ttl`은 table/source/rule 수준의 의미 계층이며, PostgreSQL `label`/feature schema를 대체하거나 생성하지 않습니다.

## 6. 데이터 파이프라인 (Data Pipeline)

로컬 `data/raw/` 및 `data/processed/` 디렉토리의 파일들을 PostgreSQL 데이터베이스로 자동 업로드 및 검증하기 위한 Python 기반 적재 파이프라인 명세입니다.

- **스크립트 경로**: `scripts/db_load_raw.py`
- **구동 방식**: `.env`의 DB 접속 정보를 읽어 도메인별 4개 로더(`krx`, `gir`, `energy_macro`, `public`)를 순차 실행.
- **예외 처리 기법**:
  1. `NaN` -> `None` 캐스팅 후 `NULL` 값으로 DB 적재 (Smallint Out of Range 방지).
  2. 숫자형 콤마(`1,000`) 파싱을 위해 Pandas `thousands=','` 적용.
  3. GIR 공공데이터의 특수문자 및 BOM(`\ufeff`) 제거 로직 포함.
  4. 멱등성 보장 (`INSERT ... ON CONFLICT DO NOTHING`).
- **정합성 검증 (`db_verify.py`)**:
  원천 테이블별 행 수, 자연키 UNIQUE 위반 여부, 주요 날짜 범위, `raw.krx_ets_daily`의 `(trd_dd, isu_code)` 중복 여부를 검증합니다. `label` 계층 검증은 라벨 설계 확정 후 재도입합니다.
